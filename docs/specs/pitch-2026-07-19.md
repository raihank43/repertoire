# Pitch — personal skills library (working name: open)

_Draft input for `/project-init`, written 2026-07-19. This is a **pitch, not a decided design** — the init brainstorm should challenge it, research the open questions, and shape the roadmap with me. Hard constraints are marked; everything else is negotiable._

## The idea

A personal library of **agent skills** — general-purpose and work-related — usable across Claude Code (and possibly Codex) sessions: at work, in side projects, and alongside the growing-docs plugin. One repo, one folder per skill, installable into an agent's skills directory.

Direct inspiration: my senior's [asrafilll/work-skills](https://github.com/asrafilll/work-skills) — its shape is proven (one `SKILL.md` per folder with YAML frontmatter, third-party skills pinned to upstream SHAs with provenance noted, an interactive installer targeting both agents, a README table with one-liners). I want my **own** library rather than contributing to theirs: different skill mix, my own conventions, and a distinctive angle (below).

## The distinctive angle — growing-docs-compliant skills

Most skills print their output to chat, where it evaporates. My projects run [growing-docs](https://github.com/raihank43/claude-growing-docs), whose integration principle is **"endings, not processes"**: a generic discipline (debugging, TDD, ideation) integrates with the doc system at the point where its *output lands* — a debug session ends by writing the root cause to a feature doc's Gotchas and the failed hypotheses as negative space; an ideation session ends by writing kept ideas to `docs/BACKLOG.md`.

So the house convention for this repo (shape to be decided in the brainstorm): **every skill that produces durable knowledge defines its docs-shaped ending** — where the output lands when the host project is a growing-docs project, and a graceful chat-only fallback when it isn't. The skill must work in *any* repo; the docs ending activates opportunistically.

This is deliberately the **other half** of a boundary growing-docs itself drew: commands whose input AND output are docs artifacts belong in the plugin (that's why `/scout` became a plugin command); generic engineering disciplines belong adopter-side — i.e., **here**.

## Hard constraints

- **Separate repo from growing-docs** — settled reasoning (two-copies-drift, plugin stays zero-dependency, doctrine is the adopter's call). This repo depends on growing-docs conventions; growing-docs must never depend on this repo.
- **Third-party skills carry provenance** — upstream repo + pinned SHA + what was modified, like the senior's repo does. Never silently fork someone else's skill.
- **Skills are prompts, not code** (markdown + frontmatter); any installer stays a thin copy script.

Second piece of evidence the repo's model is broader than one-`SKILL.md`-per-folder: this resident is a *skill + agent bundle* that must install together. _(fleshed out under [Follow-ups](#follow-ups) below.)_
- **Candidates to triage during brainstorm** (not commitments): a debug-to-gotcha skill (growing-docs has a parked backlog entry for piloting this — decide which repo owns it); pre-flight-style briefing; handoff/compaction helpers; work-specific flows (e.g. ticket/PR conventions for my job's stack).

## Open questions for the init brainstorm

1. **Name** — something that isn't a collision with the senior's `work-skills`.
2. **Install mechanism** — copy-script installer (senior's model) vs Claude Code plugin marketplace (`skills/` can ship in plugins) vs plain manual copy. Research which fits skills that must reach BOTH `~/.claude/skills/` and possibly Codex.
3. **The compliance convention's concrete shape** — a required "## Ending" section in each SKILL.md? A frontmatter field? Just a README convention? (Keep it lightweight — a heavy template would be the doc-theater failure mode.)
4. **Scope boundary with the growing-docs plugin** — write down the routing rule so future skills/commands land in the right repo without re-litigating (plugin: docs-artifacts in AND out; here: generic discipline with an optional docs ending).
5. **Work vs personal split** — one repo with folders/tags, or does work-confidential content force a private second repo? (Privacy: work skills must not leak employer specifics into a public repo.)
6. **Codex/other-agent support** — worth the variant maintenance (senior's repo has `variants/codex/`), or Claude-only until needed?

## Follow-ups

### `/minimalism` migration — captured 2026-07-19 from the design + benchmark thread

**Current state (what exists to migrate).** A user-scoped Claude Code command at `~/.claude/commands/minimalism.md`, now at **v3**. Self-contained: it embeds the ruleset and scaffolds it into a repo — it does *not* depend on the ponytail plugin, and is in fact an *alternative* to it (running both double-injects the same rules). When args are omitted it resolves three settings via `AskUserQuestion`:
- **intensity** — `lite | full | ultra` (no silent default; asked if not given)
- **scope** — `code+chat` vs `code-only` (whether the ruleset governs chat prose too; deliberately independent of intensity)
- **destination** — `.claude/rules/minimalism.md` (recommended), append to root `CLAUDE.md`, or `CLAUDE.local.md` (personal, auto-gitignored)

Idempotent: detects the `<!-- minimalism-rules v… -->` marker across all four candidate locations, replaces the block in place, consolidates duplicates. Live deployment: **setsuna**.

**The core routing decision (the line-27 question) — and the insight that answers most of it.** Minimalism is the one known resident that is **not skill-shaped, by design.** The ponytail analysis that started this thread established *why*: behavioral guardrails are badly suited to model-invocation, because the model won't reliably self-invoke discipline against an instinct it doesn't recognize it has (you don't load a "write less code" skill at the moment over-building feels like productivity). So minimalism must stay **always-on** — a rules file, not a `SKILL.md` the model chooses to read.
→ **Carry into the brainstorm:** this repo therefore needs to host a *rule + its installer command*, not only skills — the first evidence its model is broader than "one `SKILL.md` per folder" (bears on #1 name, #3 convention, #4 boundary). Working recommendation: the repo owns **both** — the ruleset text as canonical source-of-truth, and the command as delivery/installer — deploy target `.claude/rules/`, not `skills/`.

**growing-docs composition — a working exemplar of this repo's distinctive angle.** growing-docs' scaffold `CLAUDE.md` carries a *semantic* "Minimalist-ruleset carve-out": it detects an active token-efficiency ruleset **by vocabulary, not by filename**, and protects the docs' why-capture (gotchas, rejected ideas, failed hypotheses) from being minimized away. Because the minimalism block deliberately uses the "token-efficiency / shortest diff" phrasing the gate recognizes, the two **compose with zero wiring** — growing-docs stays ruleset-agnostic; minimalism is a pluggable layer it defers to.
→ Shape note for #3: minimalism does *not* produce durable docs (no "## Ending" writing to BACKLOG). Its growing-docs compliance is the **inverse** — it's the ruleset the doc-system's gate *yields to*. So "growing-docs-compliant" has at least two shapes: **produces a docs ending** (most skills) vs **recognized by a growing-docs gate** (minimalism). The convention must cover both, or scope itself to the first.

**Provenance (per the hard constraint).** Derived from **ponytail** (`github.com/DietrichGebert/ponytail`, MIT) — reimplemented, not forked (the whole product was ~40 lines of system-prompt English, so it was rebuilt rather than vendored). Changes from upstream: renamed the `ponytail:` shortcut-comment convention to `minimal:`; added an explicit comment policy (readable-code-first, comment *why* not *what*); added a **Design & UX** section + skill-invocation guard (v3); added the three-setting flow + destination routing; tuned vocabulary for growing-docs gate compatibility. Record this on the resident — honest lineage even for a reimplementation.

**Evolution + evidence (so the migration doesn't relitigate settled calls).**
- **v1** — create/append/replace-in-place file mechanics; verified by test.
- **v2** — the three-setting flow; recommended destination moved to `.claude/rules/minimalism.md` (per official Claude Code memory docs: rules files load at launch at the same priority as `CLAUDE.md`, are purpose-built for one-topic modules, and never touch a growing-docs `CLAUDE.md` — so `/project-adopt` upgrades are a non-issue).
- **v3** — design-taste preservation, driven by an A/B benchmark showing the ladder was suppressing UI/UX quality — *mainly by suppressing frontend-design skill auto-invocation, not by degrading styling ability.* Fix: a "minimize the code, never the bar" Design & UX section + a guard that invoking a relevant skill "is reading, not addition, and skipping it is not a rung" + "design bar of user-facing surfaces" added to never-simplify-away. **Deliberately intensity-independent** (taste is a boundary, not a knob).
- **Benchmark** (2026-07-08, expense-splitter, two Opus subagents; full writeup at `playground/minimalism-ab-fable/RESULTS.md`): minimalism arm matched/edged the baseline's design bar with ~26% fewer bytes, held scope (baseline built an unrequested currency selector; minimalism arm consciously deferred it), cleaner idiom, one fewer UX bug. **Caveats: n=1; and the skill-auto-invocation fix could not be exercised (subagents had no frontend-design skill) — still unverified in a real session.**

**Settled — don't re-open:** rules-file as default destination; design boundary is intensity-independent; scope (chat-prose) is orthogonal to intensity; always-on delivery (not a model-invoked skill).

**Open sub-questions for the migration:**
1. **Content storage** — does the ruleset get a canonical standalone file in the repo that the command reads, or stay embedded in the command body? (source-of-truth vs single-file simplicity)
2. **Validate the v3 skill fix** in a real session where frontend-design is installed (the benchmark couldn't).
3. **Reconcile with the parked growing-docs `BACKLOG.md` entry** (2026-07-03 batch) on minimalist-ruleset coexistence + the code-comment policy — queued for a `/forge`; decide which repo owns that decision now that this repo exists.
4. Does "a rule, not a skill" widen the repo's identity enough to affect its name and its one-folder-per-X model?

### `/orchestrate` migration — captured 2026-07-19 from the build + validation thread

**Current state (what exists to migrate).** Five user-scoped files, built and validated 2026-07-15→19, all live in `~/.claude/`:
- `skills/orchestrate/SKILL.md` — the delegation protocol (manual-invoke only via `disable-model-invocation: true`, a deliberate testing-phase choice; revisit before publishing)
- `agents/task-runner.md` — sonnet; executes a structured BRIEF verbatim; STATUS/CHANGES/VERIFICATION/NOTES report; may spawn explorers (its only permitted delegation), never splits briefs or spawns runners
- `agents/task-reviewer.md` — opus, `reasoningEffort: high`, read-only tools; independent PASS/FAIL gate that re-runs acceptance commands itself instead of trusting the runner's report
- `agents/explorer.md` — haiku, read-only; parallel fan-out retrieval scout; FINDINGS / NOT FOUND / POINTERS / FLAGS format, <150-line report cap, ~15 tool-call budget, quotes verbatim with file:line, no opinions
- `agents/advisor.md` — opus-high default, read-only, per-spawn model override for supervisor mode; RECOMMENDATION / REASONING / WOULD-CHANGE-MY-MIND / RISKS format

**The design in one paragraph.** The main-chat model orchestrates and never delegates judgment: it decomposes work, writes briefs (GOAL / FILES / CONSTRAINTS / testable ACCEPTANCE CRITERIA — if you can't write a testable criterion, the task isn't ready to delegate), routes by tier, and owns every verdict. Triage: retrieval → explorer (fired liberally, in parallel, one narrow scope each); trivial mechanical edits → runner only; substantive changes → runner + reviewer gate; judgment calls → orchestrator itself, optionally consulting the advisor. On FAIL the findings go back to the *same* runner via SendMessage (resume ≈ 2k tokens vs ≈ 20k cold spawn — measured, and the strongest empirical argument for the protocol), loop-capped at 2 rounds before the orchestrator takes over or sends the dispute to advisor arbitration. Batching rules: batch small related edits into one checklist brief; split large tasks to parallel runners only with disjoint file sets; granularity = one brief : one reviewable unit.

**Settled — don't re-open (each was deliberately decided, most tested):**
- **Flat hierarchy, no middle-management tier.** Considered (inspired by a tiered-orchestration post: K3 → GLM → DS Flash relay) and rejected: relayed briefs degrade telephone-game style, and accountability blurs — the sabotage test's clean attribution (we knew exactly who planted the bug and who caught it) is the thing a relay destroys. The post's *principles* (least privilege, unambiguous reporting lines) are kept.
- **One exception: runner→explorer is the only permitted two-level hop** — retrieval briefs ("find every caller of X") are nearly un-garblable; implementation briefs are the opposite.
- **Runners never decompose.** Decomposition is judgment; a runner whose brief feels too big reports that back (signal the orchestrator's decomposition failed) instead of improvising a split.
- **Reviewer distrusts by design.** It re-runs acceptance commands itself; taste-only nitpicks go to NOTES, FAIL requires a real defect against the brief. Judgment guidance for gray areas (requirement-serving change vs scope creep) goes *in the review brief*, not left to chance.
- **Advisor is dual-mode, declared in every brief.** MODE: PEER (advisor ≈ orchestrator tier — orchestrator owns the call, must rebut load-bearing arguments specifically to reject) vs MODE: SUPERVISOR (advisor > orchestrator, e.g. Opus main chat consulting a Fable advisor — orchestrator may NOT override on judgment/taste; disagreement = present both positions to the user and stop). Rationale from the user: the main chat won't always run the top model, and a confident orchestrator will handwave a peer away.
- **Anti-handwave rules are structural, not aspirational:** advisor's RECOMMENDATION + WOULD-CHANGE-MY-MIND relayed to the user verbatim (paraphrase is where dismissal hides); explicit user invocations ("attack this plan", "ask Fable") always honored including stance — confidence is not a reason to skip review; worst-case degrades to under-invocation the user can correct, never a silently overridden advisor.
- **Hybrid invocation triggers** — explicit always wins; automatic on: expensive-to-reverse decisions, genuinely torn, 2-round deadlock, security-sensitive, plan >~5 files.
- **Cheap-model leash** (user's standing principle: proactively watch delegated cheap models for cap/bottleneck violations): explorer got its hard cap + tool budget after validation showed haiku leaking (3× report-cap overflow on one; 8-min/42-call sweep on another despite "sample a few"). Escalation order if leaks persist: harder prompt wording → tool-call budget → model bump, in that order.

**Evidence trail (all 2026-07-16→19, summarized in memory `project-orchestrate-skill`):**
- *Smoke test:* runner+reviewer round-trip PASS on first try.
- *Sabotage test:* runner instructed to plant a defect (`", "` CSV join) behind a confident vague report — reviewer caught it at exact file:line, explicitly flagged that the runner's verification was "self-asserted, not executed"; FAIL → same-runner fix → same-reviewer re-review converged to PASS.
- *Explorer fan-out:* 3 parallel explorers mapped a real work repo (~110k tokens burned in throwaway subagent context, ~6k of synthesized report entering main context — the actual win is context preservation, not token totals); citations spot-checked accurate; NOT FOUND sections caught a real docs discrepancy.
- *Full-dress rehearsal:* seeded-wart demo project; batched 3-item runner (incl. live runner→explorer hop: found exactly the 2 rename sites) PASS; parallel disjoint-boundary runner PASS (wrote tests against the other runner's *mid-flight target behavior*, watched them flip green); both reviewer gates PASS incl. correct scope-creep-vs-requirement judgment; advisor PEER-mode first live run showed the exact behavior the design gambled on — agreed with the orchestrator's lean while catching a real reasoning flaw (torn-file vs lost-update conflation) plus a Windows `os.replace` gotcha, and independently spotted a seeded bug. No manufactured objections, no rubber stamp.
- *Honest economics:* delegation is a 10–30× token *loss* below ~5 minutes of mechanical work (the skill's triage threshold exists for this reason). It pays via main-context preservation, bulk mechanical churn on cheap models, and parallelism — not as a universal efficiency win.
